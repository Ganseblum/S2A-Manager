## 安全警告

> **🚨 以下操作可能导致数据丢失，禁止在生产环境执行：**

| 危险操作 | 命令/行为 | 后果 |
|---------|----------|------|
| ❌ 删除数据库 | `DROP DATABASE s2amanager;` | **全部数据丢失，不可恢复** |
| ❌ 重置数据库 | `prisma migrate reset` | **全部数据丢失** |
| ❌ 删除 .env 文件 | `rm .env` | API Key 无法解密，相当于丢失所有连接配置 |
| ❌ 更换 ENCRYPTION_KEY | 修改 `.env` 中的 `ENCRYPTION_KEY` | 已保存的 Admin API Key 全部无法解密 |
| ❌ 直接操作 PostgreSQL | 手动 DROP/DELETE 表数据 | 破坏数据一致性 |

> ✅ **安全原则**：任何操作前先备份，任何 `rm`/`DROP`/`reset` 命令三思而后行。

## 环境要求

| 组件 | 最低要求 | 推荐 |
|------|---------|------|
| 操作系统 | Ubuntu 20.04 / Debian 11 | Ubuntu 22.04 LTS |
| Node.js | 20.x | 22.x |
| npm | 10.x | 最新稳定版 |
| PostgreSQL | 14+ | 16 |
| 内存 | 1GB | 2GB+ |
| 磁盘 | 10GB | 20GB+ |
| 网络 | 能访问 Sub2API 管理接口 | - |

> ⚠️ **端口注意**：Sub2API 默认用 3000，S2A Manager 也默认 3000，**同一台机器部署必须改端口**。

## 前置准备

### 1. 确认 Sub2API 状态

```bash
# 确认 Sub2API 正在运行
curl http://127.0.0.1:3000/api/status

# 获取 Sub2API Admin API Key（在 Sub2API 管理后台生成）
# 记录下来，部署 S2A Manager 时需要填写
```

### 2. 确认端口不冲突

```bash
# 查看 3000 端口是否被占用
sudo lsof -i :3000
# 如果被 Sub2API 占用，S2A Manager 改用 3001 端口

# 确认 3001 端口可用
sudo lsof -i :3001  # 无输出表示可用
```

### 3. 安装 PostgreSQL（如果还没有）

```bash
# Ubuntu/Debian
sudo apt update
sudo apt install postgresql postgresql-contrib -y

# 确认 PostgreSQL 已启动
sudo systemctl status postgresql
```

## 首次部署

### Step 1：创建 PostgreSQL 数据库

```bash
# 切换到 postgres 用户
sudo -u postgres psql
```

在 PostgreSQL 交互界面执行：

```sql
-- 创建数据库用户（密码请替换为你自己的）
CREATE USER s2amanager WITH PASSWORD 'YourStrongPasswordHere';

-- 创建数据库
CREATE DATABASE s2amanager OWNER s2amanager;

-- 授权
GRANT ALL PRIVILEGES ON DATABASE s2amanager TO s2amanager;

-- 连接到新数据库，创建 schema 权限
\c s2amanager
GRANT ALL ON SCHEMA public TO s2amanager;

-- 退出
\q
```

> ✅ 安全：此步骤只创建新数据库，不影响已有数据库。

### Step 2：克隆代码

```bash
cd /opt
sudo git clone https://github.com/langrenjh-alt/S2A-Manager.git s2a-manager
sudo chown -R $USER:$USER /opt/s2a-manager
cd /opt/s2a-manager
```

> ⚠️ 如果使用 fork 仓库，请替换为你的仓库地址。

### Step 3：安装依赖

```bash
cd /opt/s2a-manager
npm ci
```

> ✅ 安全：`npm ci` 只安装 `package-lock.json` 中的依赖，不会更新版本。

### Step 4：配置环境变量

```bash
cp .env.example .env

# 生成 ENCRYPTION_KEY（重要！保存好，不能更换）
openssl rand -base64 32

# 生成 APP_SECRET
openssl rand -base64 24
```

编辑 `.env`：

```env
# PostgreSQL 连接串（替换为实际密码）
DATABASE_URL="postgresql://s2amanager:YourStrongPasswordHere@127.0.0.1:5432/s2amanager?schema=public"

# 登录会话签名（至少 24 位，替换为上面生成的值）
APP_SECRET="这里填 openssl rand -base64 24 生成的值"

# API Key 加密密钥（32 字节 base64，替换为上面生成的值，生成后不能换！）
ENCRYPTION_KEY="这里填 openssl rand -base64 32 生成的值"

# （可选）S2A Manager 监听端口，默认 3000
# 如果与 Sub2API 冲突，改为 3001
PORT=3001
```

> ⚠️ **重要**：`ENCRYPTION_KEY` 生成后**绝对不能更换**，否则已保存的 Admin API Key 无法解密！
> 建议把 `.env` 文件内容备份到安全的地方。

### Step 5：初始化数据库

```bash
npx prisma migrate deploy
```

> ✅ 安全：此命令只创建表结构，不会删除任何数据。
> ⚠️ 如果报错，不要执行 `prisma migrate reset`，先查看错误信息。

### Step 6：构建

```bash
npm run build
```

### Step 7：启动测试（先不用 systemd）

```bash
# 终端 1：启动 Web
npm run start

# 终端 2：启动 worker
npm run worker
```

访问 `http://服务器IP:3001`，应该看到初始化页面。

### Step 8：创建管理员账号

1. 浏览器访问 `http://服务器IP:3001`
2. 首次访问自动跳转到初始化页面
3. 填写管理员邮箱、密码（至少 6 位）
4. 提交后自动跳转到登录页
5. 用刚创建的账号登录

### Step 9：测试 Sub2API 连接

1. 登录后点击「添加连接」
2. 填写：
   - 名称：随便起（如「我的 Sub2API」）
   - Sub2API 站点 URL：`http://127.0.0.1:3000`（如果同一台）
   - Admin API Key：填写你的 Sub2API Admin API Key
3. 点「测试」确认能连通
4. 保存

### Step 10：用 systemd 托管（生产环境）

创建 Web 服务：

```bash
sudo nano /etc/systemd/system/s2a-manager-web.service
```

粘贴以下内容：

```ini
[Unit]
Description=S2A Manager Web
After=network.target postgresql.service

[Service]
Type=simple
User=你的用户名
WorkingDirectory=/opt/s2a-manager
Environment=NODE_ENV=production
Environment=PORT=3001
ExecStart=/usr/bin/npm run start
Restart=always
RestartSec=5
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

创建 worker 服务：

```bash
sudo nano /etc/systemd/system/s2a-manager-worker.service
```

粘贴以下内容：

```ini
[Unit]
Description=S2A Manager Worker
After=network.target postgresql.service

[Service]
Type=simple
User=你的用户名
WorkingDirectory=/opt/s2a-manager
Environment=NODE_ENV=production
Environment=S2A_WORKER_INTERVAL_SECONDS=600
ExecStart=/usr/bin/npm run worker
Restart=always
RestartSec=5
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

> ⚠️ 把 `你的用户名` 替换为实际运行 `npm` 的系统用户名（用 `whoami` 查看）。

启用并启动服务：

```bash
sudo systemctl daemon-reload
sudo systemctl enable s2a-manager-web s2a-manager-worker
sudo systemctl start s2a-manager-web s2a-manager-worker
sudo systemctl status s2a-manager-web s2a-manager-worker
```

### Step 11：验证运行状态

```bash
# 查看服务日志
sudo journalctl -u s2a-manager-web -f
sudo journalctl -u s2a-manager-worker -f

# 测试 HTTP 响应
curl -I http://127.0.0.1:3001

# 在浏览器访问，进入「服务状态」页面，确认 worker 在线
```

## 备份策略

> 📌 **备份频率建议**：
> - `.env` 文件：**立即备份**（包含加密密钥，丢失无法恢复）
> - 数据库：**每天一次**（cron 自动备份）
> - 代码目录：**每次升级前**

### 备份 .env 文件（最重要！）

```bash
# 备份到安全位置（不在项目目录下）
cp /opt/s2a-manager/.env ~/.s2a-manager-env-backup

# 或者加密备份
gpg -c /opt/s2a-manager/.env  # 会生成 .env.gpg，记住密码
```

> ⚠️ **ENCRYPTION_KEY 丢失 = 所有已保存的 Admin API Key 无法解密 = 需要重新填写所有连接配置**

### 备份 PostgreSQL 数据库

```bash
# 每天自动备份脚本
sudo nano /opt/backup-s2a-manager.sh
```

脚本内容：

```bash
#!/bin/bash
# S2A Manager 数据库备份脚本
BACKUP_DIR="/opt/backups/s2a-manager"
DATE=$(date +%Y%m%d_%H%M%S)
FILENAME="s2amanager_${DATE}.sql.gz"

mkdir -p "$BACKUP_DIR"

# 备份数据库
pg_dump -h 127.0.0.1 -U s2amanager s2amanager | gzip > "$BACKUP_DIR/$FILENAME"

# 只保留最近 7 天的备份
find "$BACKUP_DIR" -name "*.sql.gz" -mtime +7 -delete

echo "备份完成: $BACKUP_DIR/$FILENAME"
```

设置定时执行：

```bash
chmod +x /opt/backup-s2a-manager.sh

# 添加到 cron（每天凌晨 2 点备份）
(crontab -l 2>/dev/null; echo "0 2 * * * /opt/backup-s2a-manager.sh >> /opt/backups/s2a-manager/backup.log 2>&1") | crontab -
```

手动执行一次测试：

```bash
/opt/backup-s2a-manager.sh
ls -lh /opt/backups/s2a-manager/
```

### 备份整个项目目录（升级前必做）

```bash
# 升级前完整备份
sudo cp -a /opt/s2a-manager /opt/s2a-manager.backup.$(date +%Y%m%d_%H%M%S)

# 列出备份
ls -lh /opt/s2a-manager.backup.* 
```

## 升级操作

> ⚠️ **升级前必做**：
> 1. 完整备份项目目录（`cp -a`）
> 2. 备份数据库（`pg_dump`）
> 3. 备份 `.env` 文件
> 4. 在测试环境先验证新版本

### 标准升级流程

```bash
# 1. 停止服务
sudo systemctl stop s2a-manager-web s2a-manager-worker

# 2. 备份（重要！）
sudo cp -a /opt/s2a-manager /opt/s2a-manager.backup.$(date +%Y%m%d_%H%M%S)
pg_dump -h 127.0.0.1 -U s2amanager s2amanager | gzip > ~/s2amanager_backup_$(date +%Y%m%d).sql.gz
cp /opt/s2a-manager/.env ~/

# 3. 拉取新代码
cd /opt/s2a-manager
git pull --ff-only
# 如果有冲突，不要强制合并，先检查
# 如果 pull 失败，用备份恢复，排查问题

# 4. 安装新依赖
npm ci

# 5. 数据库迁移（只执行新增迁移，不会删除数据）
npx prisma migrate deploy

# 6. 重新构建
npm run build

# 7. 启动服务
sudo systemctl start s2a-manager-web s2a-manager-worker

# 8. 确认服务正常
sudo systemctl status s2a-manager-web s2a-manager-worker
curl -I http://127.0.0.1:3001

# 9. 查看日志确认无报错
sudo journalctl -u s2a-manager-web -n 50
sudo journalctl -u s2a-manager-worker -n 50
```

### 升级失败回滚

```bash
# 停止服务
sudo systemctl stop s2a-manager-web s2a-manager-worker

# 恢复代码
LATEST_BACKUP=$(ls -td /opt/s2a-manager.backup.* | head -1)
sudo rm -rf /opt/s2a-manager
sudo mv "$LATEST_BACKUP" /opt/s2a-manager

# 恢复 .env
cp ~/.s2a-manager-env-backup /opt/s2a-manager/.env

# 重启服务
sudo systemctl start s2a-manager-web s2a-manager-worker
```

## 回滚操作

### 数据库回滚（如果需要）

```bash
# 恢复数据库（会覆盖当前数据，确保这是你想要的）

# ⚠️ 此操作会覆盖现有数据，执行前再次确认备份文件正确
```

### 代码回滚到指定版本

```bash

git log --oneline -10

# 回滚到指定 commit
git checkout <commit-hash>

# 重新安装依赖并构建
npm ci
npm run build

# 重启服务
sudo systemctl restart s2a-manager-web s2a-manager-worker
```

## 常见问题

### Q1：部署后访问不了页面

```bash
# 检查服务是否运行
sudo systemctl status s2a-manager-web

# 检查端口是否监听
sudo lsof -i :3001

# 检查防火墙
sudo ufw status
sudo ufw allow 3001/tcp
```

### Q2：Sub2API 连接测试失败

- 确认 Sub2API 正在运行：`curl http://127.0.0.1:3000`
- 确认 Admin API Key 正确（在 Sub2API 管理后台重新生成）
- 确认 URL 格式正确（不要加 `/api`，如 `http://127.0.0.1:3000`）

### Q3：worker 不运行

```bash
sudo systemctl status s2a-manager-worker
sudo journalctl -u s2a-manager-worker -n 50
```

常见原因：
- PostgreSQL 未启动
- `ENCRYPTION_KEY` 不对，无法解密 API Key
- 数据库连接失败，检查 `DATABASE_URL`

### Q4：升级后页面报错

```bash
# 重新构建
cd /opt/s2a-manager
npm run build
sudo systemctl restart s2a-manager-web
```

### Q5：如何确认 S2A Manager 不会影响 Sub2API

- S2A Manager 只通过 Admin API 与 Sub2API 通信，不可以直接修改 Sub2API 文件或数据库
- 倍率同步功能默认不启用，需要手动开启
- 建议先在 Sub2API 建一个测试分组，用 S2A Manager 绑定并测试
